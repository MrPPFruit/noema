# Noema UI 开发规范

本文定义 Noema 当前 UI 开发的默认规范。所有新页面、新组件、视觉精修和交互迭代都应遵守本规范。

## 0. 当前状态

用户已确认 Flutter Home / `境`、Import / `入`、Observe / `观` 和 Cull / `甄` 的主体验基线。Home 进入本地保存的 `境`；Import 创建或追加照片；Observe 查看全部照片、编辑境名、追加照片、多选移除，并提供单张照片查看器和 `甄 / 赏 / 鉴` 后续入口；Cull / `甄` 支持快甄、对照甄、缩略图召回、单图 / 双图预览和可逆决策。当前底部主动作已统一为方形玻璃签语言，`观` 的后续入口已收敛到 `intent-seal` 意图轴。当前阶段不要另起 HTML 迁移实现，也不要为了追求像素级重做而偏离 Flutter-first 主线。后续如需调整这些页面，应先更新设计规格或 OpenSpec 任务，再做最小范围变更。

## 1. 默认开发流程

Noema UI 开发采用 Flutter-first 流程：

1. 先判断是否需要使用 Comet / OpenSpec；涉及产品行为、页面职责、功能边界或多文件实现时默认进入 Comet。
2. 先阅读 `docs/design/ui-design-spec.md`、`docs/design/noema-vi.md` 和相关页面规格，确认页面的视觉方向、交互规则和已批准的 VI 原则。
3. 制作 UI 前检查 GSAP 动效参考和 COSS UI 组件参考，并把适用结论写入页面规格或设计文档。
4. 直接在 Flutter 中实现 UI，不再先制作独立 HTML 原型再迁移。
5. 使用 Flutter Web 作为快速预览与视觉迭代入口。
6. 视觉确认后，再使用 Android `emulator-5556` 或真机做平台复核。
7. 完成后同步检查相关功能逻辑和文档索引，避免规范、设计和实现脱节。

HTML 仅可作为历史参考或临时设计说明，不应再作为可迁移实现层。

## 2. 为什么不用 HTML 作为中间实现

独立 HTML 原型会引入另一套运行时差异：

- CSS 字体 fallback 与 Flutter / Android 字体 fallback 不一致。
- SVG / Canvas / Flutter painter 的坐标系和缩放规则不同。
- CSS mask、blend mode、backdrop-filter、阴影和字体渲染与 Flutter 存在差异。
- HTML 看起来正确，不代表 Flutter App 中同样正确。

因此，从当前阶段开始，UI 迭代应在最终技术栈 Flutter 中完成。禁止把 HTML 原型当作“先实现、再迁移”的主流程；如确需保留 HTML，只能放在设计参考或历史说明中，并明确它不是实现来源。

## 3. Flutter Web 预览

普通 UI 精修优先使用 Flutter Web 浏览器预览：

```text
flutter run -d chrome
```

如需固定端口，使用：

```text
flutter run -d web-server --web-hostname 127.0.0.1 --web-port 5173
```

Flutter Web 预览用于快速确认：

- 字体是否来自已打包资产。
- 图标尺寸、坐标和视觉中心是否正确。
- 页面布局、间距、层级、背景和动效是否接近设计规范。
- 设置面板、列数切换、排序、长按状态等交互是否可用。
- 页面在浏览器中是否可快速复现问题，便于截图、对照和迭代。

## 4. 移动端复核

Flutter Web 通过后，再做移动端复核。当前项目约定：

- 不再使用 `emulator-5554`，避免与其他项目抢占资源。
- 默认使用 `emulator-5556` 做 Android 模拟器验证。
- 如需复核真实设备问题，优先 Android / Samsung 设备。

移动端复核重点：

- 系统状态栏和导航栏影响。
- 安全区、屏幕密度、真实触摸区域。
- 打包字体、图片资源和解码行为。
- 首屏启动、滚动性能和平台差异。
- Home 页已确认后的回归风险，尤其是顶部 wordmark、中文标题、工具图标和网格区域的对齐。

## 5. 字体规范

中文视觉字体：

- 使用 `Luo`。
- Flutter 字体资产：`assets/fonts/Luo-Regular.ttf`。
- `assets/fonts/Luo-Regular.woff2` 仅作为 HTML / Web 原型参考资产保留；Flutter 原生和 Web 构建都必须注册 ttf，避免 Android 回退系统中文字体。
- 只要可见文本或 tooltip 包含中文，就必须显式使用 `Luo`，包括照片数量里的 `张`、一字功能入口和轻提示。
- Flutter 全局 Theme 必须保留 `fontFamilyFallback: ['Luo']`，确保系统 Tooltip、旧占位页或遗漏显式字体的中文在 Android 上仍优先使用 Luo。
- 中文不应依赖系统 fallback 来达成设计效果。

英文 wordmark 字体：

- 使用 `NoemaLatin`。
- 字体资产：`assets/fonts/CormorantGaramond.ttf`。
- 来源为 Google Fonts 的 Cormorant Garamond，OFL 授权。
- 用于 `Noema` wordmark，避免依赖 Android / iOS / macOS 系统字体 fallback。
- 文档中可写作 `NoemaLatin / Cormorant Garamond`，但 Flutter 代码里应优先引用已注册的 `NoemaLatin` 字体族。

字体检查要求：

- Flutter Web 预览必须确认字体来自打包资产，而不是系统默认字体。
- Android `emulator-5556` 复核必须确认英文 wordmark 与中文展示文字没有 fallback 走样。
- 设计规格、开发规范和实现中的字体命名必须保持一致。

## 6. 图标规范

- 图标实现必须以 Flutter 最终渲染为准。
- 若从 SVG 迁移图标，必须保留 SVG 的 viewBox 坐标语义，再缩放到 Flutter 实际画布。
- 不能直接把 `viewBox 0 0 24 24` 的坐标画到 `22 × 22` 画布中，否则图标会偏心。
- 所有工具图标应放入固定尺寸按钮盒，并按按钮盒居中。
- 同一工具栏中的图标按钮应使用一致的外框尺寸、内边距、命中区域和视觉中心。
- 图标、文字、网格和面板之间的间距以设计规格为准；若 Flutter 最终渲染与规格存在差异，优先修正 Flutter 布局，而不是回到 HTML 原型。

## 7. 布局对齐规范

- 核心页面必须共享统一手机画布规则。Flutter Web 预览中，Home / `境`、Import / `入`、Observe / `观` 等页面应保持同一最大宽度、居中方式和背景边界。
- 核心页面必须复用统一 scene metrics 和 palette。不得在单个页面内重新硬编码顶部高度、安全区偏移、背景渐变或色调色值，除非页面规格明确批准。
- `Noema` wordmark 默认作为顶部居中品牌锚点；左右图标按钮不得把它挤偏。若右侧有工具、左侧无工具，应通过 Stack / 占位保持视觉中心。
- 返回、设置、取消等顶部图标按钮的视觉盒必须与页面主内容边界对齐；为了触控可让点击热区略微外扩，但玻璃背景或可见圆形不能比内容列更靠内。
- `Noema` wordmark 的顶部坐标和中心点属于回归项；新增或改动核心页面时应通过测试或截图确认 Home / `境` 与目标页保持一致。
- 背景主题字必须使用共享 `NoemaThemeMark`。由于不同汉字的真实字形边界不同，必要的视觉校正应沉淀在组件内部，禁止在页面里临时调整 `left/top/fontSize` 来追齐或导致裁切。
- 主动作默认使用底部方形玻璃签按钮语言。创建、添加、确认、一字入口等同层级动作应保持视觉卡片尺寸、命中区、玻璃表面、阴影、纹样和图标 / 字心中心一致。
- 已有共享组件时必须复用，例如 Home 创建、Import 添加照片、Import 完成创建、Import 预览关闭、Observe 空态添加和 Observe `甄 / 赏 / 鉴` 入口都应使用 `NoemaSquareActionButton` 或委托到它的兼容封装，不能临时复制出形态不同的按钮。
- 空态不为了对称而显示无意义确认按钮；没有照片时 Import 只保留添加入口，导入照片后再显示完成入口。
- 对于“有上下文但缺少必填项”的动作，可以低亮保留并提供点击反馈，例如 Import 有照片但无境名时点击完成入口应提示命名，而不是静默无效。低亮可点不是不可见态，light / dark 模式下仍要保证图标、边框和方形轮廓清晰。
- 选择态工具应靠近被操作内容。Import 多选照片时，用取消和删除图标替换目的文案/计数行，不把选择态文字挤进顶部品牌行。
- Observe / `观` 的照片墙是页面主体，境名下方的信息行承载照片数量、时间排序、导入和密度切换；这些工具不应被塞进右上角更多菜单。`观` 首版排序只做时间正序 / 时间倒序，暂不做名称或修改时间排序。时间排序图标必须让方向箭头独立可读，不能叠进日历格子、时钟轮廓或其他复杂图形内部；组合后的视觉重量要接近同排 24px 工具图标。
- Observe / `观` 的照片墙采用稳定自然比例 masonry 布局，保留横竖图自然比例；同一密度下列宽稳定，高度由真实宽高决定。密度切换和双指缩放应吸附到离散级别，并用稳定 id 做自然重排动画；大批量照片需降级重位移动效并保持可视区域构建。照片墙是主视觉，底部边缘应参考 Home / `境` 用短距离渐隐融入场景，不得硬切，也不得使用明显暗遮罩压缩观看面积；左右绘制层必须保留阴影安全边距，避免窄屏 Android 裁切边缘照片阴影。
- Observe / `观` 底部功能入口为 `甄 / 赏 / 鉴`，只作为体验入口，不自动触发旧 review、分析、评分或整理流程。当前确认方案为 `intent-seal`：`赏` 默认吸附在中间，点击中签触发沉浸欣赏；拖到可用的 `甄` / `鉴` 后中心签替换为对应字，松手触发并自动回中。两侧能力不可用时不显示，可用时才以低亮签文、隐章角标、淡虚线和从 `赏` 向两侧传递的脉冲提示，不使用生硬横线、强流光或移动亮条。`赏` 的左右终点必须能与侧向目标中心重叠；`?dock=intent-seal&tune=1` 仅开放左右终点数值输入用于校准，不作为普通用户功能。不要把三者做成完全同级的常驻底部导航，也不使用厚重的整条导航胶囊或“图标 + 字”工具按钮组合。入口应轻覆盖在照片墙和场景上，照片墙滚动窗口参考 Home / `境` 延展到画布底部，让底部渐隐位置更低。
- `甄` 的相似组 / 连拍组发现应在 Import 创建 / 追加完成后的本地精筛结果中形成，而不是等用户点击 `甄` 后才开始扫描。当前不做应用启动时全图库后台筛选，也不要求进入 `观` 后再低优先级补筛；`甄` 只消费当前 workspace 已有可处理组并承载处理流程。
- Cull / `甄` 的快甄和对照甄必须保持上下决策方向一致：向上为出境 / 丢弃，向下为保留。缩略图召回必须清除原决策并把照片带回中心，不能只做视觉撤销。对照甄只剩最后一张时应按单图居中处理，不显示空占位线框。
- `甄` 的大图预览不显示额外关闭或重置按钮。退出方式是系统返回或点击图片外区域；双图预览应同步缩放 / 平移，并用轻量轨迹或贴边描边表达来源关系。
- Observe / `观` 点击照片进入全屏照片查看器。查看器必须完整显示照片，按需加载更大预览，支持左右滑动、双击放大 / 复位、双指缩放和缩小退出；放大后应锁定横向翻页，避免拖拽冲突。
- Observe / `观` 的境名可原地编辑：默认态只显示低权重编辑图标，编辑图标放在标题行右侧；编辑态才显示 10 字限制计数、保存和取消入口；不使用长下划线。
- Observe / `观` 长按照片进入多选移除状态时，隐藏境名下方的照片墙工具行，显示取消和移除入口；移除只作用于当前 `境`，原照片仍在系统相册中。
- 从 Observe / `观` 添加照片进入 Import / `入` 时必须使用追加态，不显示空命名输入，只显示当前境名、已有数量和本次新增照片。
- Android 系统返回必须走业务后退：照片查看器回 `观`，`观` 回 Home，追加导入回 `观`。只有 Home 首次返回显示上半部统一小胶囊提示，短时间内第二次返回才退出应用。
- Home 页顶部区域应保持 wordmark、中文视觉字、工具图标和内容网格的整体对齐。
- 固定格式元素必须有稳定尺寸，避免 hover、选中、加载、长文本或图标状态导致布局跳动。
- 文字不得和按钮、图标、卡片、网格或安全区重叠。
- 移动端复核时必须检查状态栏、导航栏和安全区对首屏布局的影响。
- 浏览器预览通过不等于移动端通过；涉及首屏、字体、触控区域和平台 chrome 的变更必须在 `emulator-5556` 上复核。

## 8. UI 参考检查规范

每次制作新页面、新组件或重要动效前，都要先做两项参考检查：

- GSAP：用于寻找动效模式参考，尤其是 page transition、shared layout、stagger、Flip 等思路。Flutter 实现不直接依赖 GSAP，但要借鉴其动效原则并记录适用结论。
- COSS UI：用于检查是否有可借鉴的组件职责和状态结构，例如 Field、Input、Empty、Scroll Area、Dialog、Drawer、Icon Button 等。Noema 不直接复制 COSS 视觉，而是参考组件分层。

结论应写入对应页面规格、OpenSpec design 或技术设计文档。后续 UI 制作默认遵守，不需要额外提醒。

## 9. Comet / OpenSpec 规范

项目变更默认遵循 Comet 工作流：

- OpenSpec 负责记录 WHAT：proposal、design、delta spec、tasks。
- Superpowers / 实施计划负责记录 HOW。
- UI 变更必须保持 `docs/design/ui-design-spec.md`、相关 OpenSpec change 和实现同步。
- 每完成一个明确任务，应同步更新 `tasks.md`。
- 验证通过后再进入归档。
- 如果只是整理文档索引或开发规范，也必须先判断是否需要 OpenSpec；不涉及产品行为变化时可只更新文档，不新增 change。
- 如果已有 active change 且当前整理属于该 change 的收尾事实，应把文档任务纳入当前 change，而不是另起重复 change。

## 10. Dirty worktree 协作规范

当前仓库可能同时存在其他执行者的未提交改动。执行 UI 或文档任务时必须遵守：

- 修改前先看 `git status --short`，确认哪些文件已经 dirty。
- 只修改任务明确授权的文件。
- 不回滚、不覆盖、不格式化与当前任务无关的文件。
- 如果必须编辑已有 dirty 文件，应在现有内容上增量整理，不把文件重写成与当前任务无关的新结构。
- 发现设计、OpenSpec、代码或测试之间存在不一致时，先记录到对应文档或任务，再按授权范围处理。
